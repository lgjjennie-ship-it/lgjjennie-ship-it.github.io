# Uclaw 需求验证调研方案（v3）

> **项目背景**：产品经理参考 ShopClawMart 业态，计划在 Compshare 下做一个类似平台「Uclaw」——AI Agent 技能/角色的交易市场。需要快速验证存量客户是否有此需求。
> 
> **调研目标**：通过了解客户**当下真实行为**（而非未来意愿），判断市场是否存在。
> 
> **核心原则**：不问"你想不想买"，只问"你现在在做什么、花了多少、踩了什么坑"。
> 
> **时间要求**：总周期 **5 个工作日**，出结果。

---

## 一、调研逻辑：从行为推需求

传统问法（❌）：你会不会买 Uclaw 的产品？
→ 得到的是**态度数据**，用户会高估自己的付费意愿

正确问法（✅）：你现在在用什么？花了多少钱？为什么花？哪里不爽？
→ 得到的是**行为数据**，用真实消费行为反推需求强度

**推导链条**：
```
当下行为 → 已验证需求 → 现有方案的痛点 → Uclaw 的切入机会
```

---

## 二、验证框架：四个关键问题

| # | 要回答的问题 | 判断标准 |
|---|-------------|---------|
| Q1 | 客户现在是否在用 AI Agent？渗透率多高？ | ≥30% 在用 → 市场存在 |
| Q2 | 在用的客户为此花了多少钱？花在哪里？ | 有实际付费行为 → 需求真实 |
| Q3 | 客户在 token 成本控制上做了什么？ | 有主动优化行为 → 痛点真实 |
| Q4 | 没在用的客户，卡在哪里？是没需求还是有障碍？ | 有障碍 → 有机会；没需求 → 放弃 |

**Q1+Q2 验证"需求是否存在"，Q3+Q4 验证"痛点是否成立"**

## 四、深度访谈提纲（Day 3-4，6-8 人）

> **方法论**：基于 JTBD（Jobs To Be Done）框架设计。核心原则：**围绕具体事件展开，不问抽象态度**。
> 访谈链路：**情境重建 → Trigger 事件 → 考虑过程 → 决策 → 使用体验 → 结果评估**

### 抽样策略

| 用户类型 | 人数 | 筛选条件 | 访谈重点（JTBD 视角） |
|---------|------|---------|---------|
| 高消费 Agent 用户 | 2-3 人 | Q4≥¥500/月 | 是什么 job 让你"雇用"了 Agent？竞争方案是什么？ |
| 有成本控制行为的用户 | 2 人 | Q6 选了 ≥2 项 | Switch 逻辑：从贵方案→便宜方案的四力分析 |
| 弃用者 | 1-2 人 | Q2=B | "解雇"Agent 的 trigger 是什么？现在"雇用"了什么替代方案？ |
| 有障碍的观望者 | 1 人 | Q8c≠"不需要" | 焦虑和惯性具体是什么？被动寻找→主动寻找的转折点在哪？ |

---

### 访谈提纲 A：在用用户（20 分钟）

#### 模块一：热身（2 分钟）

```
1. 简单介绍一下你的工作，平时用 Compshare 主要做什么？
```
【研究员注】轻松开场，建立信任。记住对方的具体业务场景，后续追问要锚定到这个场景。

#### 模块二：情境重建——"雇用"那一刻（5 分钟）

```
2. 能帮我回想一下，你第一次决定用 AI Agent 的那个时刻吗？当时你在做什么？
3. 在用 Agent 之前，你是怎么处理这件事的？（具体方式、花多少时间）
4. 那个旧方式，当时哪里让你觉得「不行了，得换个办法」？
```
【研究员注】第 2 题锚定具体事件，不接受"我觉得 Agent 很有用"这种泛泛而谈。追问细节：**哪一天、在哪、当时手上在干什么**。第 3 题挖出竞争方案（可能是 Excel、手写脚本、甚至找实习生干），第 4 题挖推力。

#### 模块三：Trigger 与寻找过程（5 分钟）

```
5. 「觉得不行了」到「真正动手去找 Agent」中间隔了多久？是什么事让你从想想变成了行动？
6. 你当时找了哪些方案？怎么找的？（朋友推荐、搜索、看到广告…）
7. 这些方案你最后怎么选的？什么让你决定用现在这个？
```
【研究员注】第 5 题挖被动焦虑→主动寻找的转折点（trigger event）。第 6 题识别信息渠道和竞争方案全景。第 7 题挖拉力——新方案吸引他的到底是什么。

#### 模块四：四力分析——Switch 逻辑（5 分钟）

```
8. 切换到 Agent 的过程中，有没有什么让你犹豫的？差点没用上的？
9. 有没有什么让你想继续用老方法的理由？
10. 你现在每个月在 Agent 上花 [引用 Q4 金额]，花钱的时候有没有心疼过？觉得花在了什么上面？
```
【研究员注】第 8 题挖焦虑（切换成本、学习门槛、不确定性）。第 9 题挖惯性（旧方案的舒适区）。第 10 题是成本感知的真实感受，不是问"贵不贵"，而是"花钱那一刻你在想什么"。

**四力记录模板**（研究员在访谈中实时填写）：

| 力量 | 内容 |
|-----|-----|
| **推力**（离开旧方案的原因） | |
| **拉力**（Agent 吸引我的地方） | |
| **焦虑**（切换时的顾虑） | |
| **惯性**（留在旧方案的理由） | |

#### 模块五：使用体验与未满足 Job（3 分钟）

```
11. 用了一段时间后，它帮你解决了你最开始想解决的问题吗？
12. 如果明天 Agent 工具全部消失了，你会怎么办？你会回到之前的方式吗？
13. 现在用 Agent 过程中，有没有什么事你觉得「应该能做到但还差点意思」的？
```
【研究员注】第 11 题检验 job 是否被满足。第 12 题测依赖度（如果答案是"无所谓"，说明 job 不够强）。第 13 题挖未满足的 job——这就是 Uclaw 的机会切入点。

### 访谈提纲 B：弃用用户（15 分钟）

> **JTBD 视角**：用户曾"雇用"Agent 来完成某个 job，后来把它"解雇"了。我们要搞清楚：job 消失了？还是 Agent 没完成 job？还是找到了更好的"员工"？

#### 模块一：情境重建——"雇用"那一刻（3 分钟）

```
1. 你当时开始用 Agent 的时候，正在做什么？想解决什么具体问题？
2. 在用 Agent 之前，你怎么处理那个问题的？
```
【研究员注】先还原当初的 job，才能理解后来为什么"解雇"。

#### 模块二：从"雇用"到"解雇"（5 分钟）

```
3. 用了多长时间后你开始觉得不对劲？能说一个具体的例子吗——哪一次让你觉得「算了」？
4. 在彻底放弃之前，你有没有试过其他方法让它好用？（调 prompt、换模型、找别人配…）
5. 最后决定放弃那一刻，是因为一个具体事件，还是慢慢累积的失望？
```
【研究员注】第 3 题锚定"解雇"事件。第 4 题了解用户投入了多少沉没成本（投入越多、放弃越值得研究）。第 5 题区分 trigger 类型——突发还是渐进。

#### 模块三：竞争方案——现在"雇用"了谁（5 分钟）

```
6. 你现在怎么处理当初想用 Agent 做的那些事？
7. 现在这个方式，比 Agent 好在哪？差在哪？
8. 如果有人帮你把 Agent 配好了、调试好了、直接用——你会重新考虑吗？为什么？
```
【研究员注】第 6 题挖出现在的竞争方案。第 7 题做四力分析（新方案 vs Agent 的推力拉力）。第 8 题测试的不是"意愿"，而是"焦虑和惯性的阈值"——如果"开箱即用"能消除焦虑，说明 job 还在，只是 Agent 没满足好。

**弃用者四力记录模板**：

| 力量 | 内容 |
|-----|-----|
| **推力**（离开 Agent 的原因） | |
| **拉力**（替代方案吸引我的地方） | |
| **焦虑**（回去用 Agent 的顾虑） | |
| **惯性**（留在替代方案的理由） | |

#### 收尾（2 分钟）

```
9. 回头看，你觉得 Agent 类工具要变成什么样，才能真正帮到你做的事？
```
【研究员注】开放题，让用户描述理想状态。注意区分：用户说的是 functional job（功能）还是 emotional job（感受）。

### 访谈提纲 C：观望用户（15 分钟）

> **JTBD 视角**：用户可能有未被满足的 job，但还没有"雇用"Agent。焦虑和惯性阻止了 switch。也可能根本没有对应的 job。

#### 模块一：当下情境（5 分钟）

```
1. 你日常工作中最耗时间、最重复的事是什么？能举一个上周的例子吗？
2. 你现在怎么处理这些事？用了什么工具或方法？
3. 这个方式有没有让你不爽的地方？
```
【研究员注】不提 Agent，先挖 job。第 1 题寻找潜在的 functional job。第 2 题找到当前的"雇员"（现有方案）。第 3 题找推力——如果没有任何不爽，说明没有 switch 动力。

#### 模块二：Agent 认知与障碍（5 分钟）

```
4. 你第一次听说 AI Agent 是什么场景？当时什么感觉？
5. 有没有过哪一刻你想「要不试试 Agent」？是什么让你这么想的？
6. 但最后没动手——当时具体卡在哪了？
```
【研究员注】第 4 题还原信息接触情境。第 5 题找"被动寻找"信号（有兴趣但没行动）。第 6 题挖焦虑和惯性——是「不知道怎么开始」（能力焦虑）、「怕浪费钱」（成本焦虑）、还是「现在也能凑合」（惯性）？

#### 模块三：假设性试探（5 分钟）

```
7. 如果现在有一个 Agent，能帮你做 [引用第 1 题提到的重复工作]，你会怎么评估要不要用？
8. 你会看什么信息来判断？（价格、案例、朋友推荐、试用？）
9. 你觉得要花多少钱/多少时间，你就觉得"不值得折腾"了？
```
【研究员注】第 7 题不是问"你想不想"，而是问"你怎么判断"——挖决策逻辑。第 8 题了解信息渠道和信任来源。第 9 题量化焦虑阈值——这是 Uclaw 定价和 onboarding 设计的关键输入。

### 访谈通用注意事项

> 💡 **JTBD 访谈核心原则**：
> 1. **永远跟着事件走，不跟着观点走**——用户开始发表意见时，温和拉回到「那一次具体经历」
> 2. **竞争方案 ≠ 同类产品**——Excel、Python 脚本、找实习生、甚至"忍着不做"都是竞争方案
> 3. **Job 是动机，不是功能需求**——"我要一个能自动发邮件的"是功能；"我不想每天花 2 小时干重复的事"是 job
> 4. **推论与证据分离**——访谈记录中明确标注「用户原话」和「研究员解读」
> 5. **每个核心发现至少对应一处原话**——不接受没有原文支撑的"洞察"

## 五、数据分析框架

### 问卷分析重点

| 分析维度 | 数据来源 | 关键指标 |
|---------|---------|---------|
| **市场渗透率** | Q2 分布 | 在用比例、弃用比例、观望比例 |
| **付费现状** | Q4 + Q5 | 月均花费、钱花在哪里 |
| **成本敏感度** | Q6 | 有多少人在主动控制成本、用什么方法 |
| **核心痛点** | Q7 / Q8 / Q8c | 痛点排名、弃用原因、观望障碍 |
| **场景需求** | Q1 × Q3 交叉 | 哪类 Compshare 用户更可能用 Agent |

### Go/No-Go 判断矩阵

```
                    付费行为真实（Q4 有实际花费）
                    是              否
渗透率高    ┌──────────────┬──────────────┐
(Q2≥30%在用)│   强 Go      │  有条件 Go   │
            │ 有人有需求   │ 有人但没找到  │
            │ 且在花钱     │ 付费方式      │
            ├──────────────┼──────────────┤
渗透率低    │   Niche Go   │   No-Go      │
(Q2<30%在用)│ 小众但高价值 │  市场不存在   │
            │ 做垂直切入   │  或时机未到   │
            └──────────────┴──────────────┘
```

### 交叉验证：后台数据

| 分析项 | 数据来源 | 验证什么 |
|-------|---------|---------|
| Modelverse API 中 Agent 相关模型调用量 | API 日志 | 和问卷 Q2 的渗透率是否吻合 |
| 长上下文/多轮对话 API 调用占比 | API 日志 | Agent 典型特征是多轮+长上下文 |
| 客户工单中 Agent/自动化相关问题 | 工单系统 | 是否有人在主动问"怎么搭 Agent" |

### 最终报告结构

```
1. 执行摘要（一页纸结论）
2. 市场现状：Agent 渗透率 + 付费规模
3. 客户画像：谁在用、用什么、花多少
4. JTBD 洞察：核心 Job Statements + 情境卡片
5. 四力分析：推力/拉力/焦虑/惯性全景图
6. 竞争方案地图：Agent vs 所有替代方案
7. 弃用/观望分析：障碍在哪里
8. 机会判断：Go/No-Go + 切入方向建议
9. 附录：原始数据 + JTBD 访谈结构化记录
```
